# 需求澄清清单

在做架构选型前，先用此清单收集足够上下文。

## 问题与结果

- 当前到底有什么问题尚未解决？
- 主要用户或系统消费者是谁？
- 交付后预期会发生什么行为变化？
- 用什么指标判断是否成功？

## 范围与约束

- V1 必须包含哪些能力？
- 明确不做的范围是什么？
- 固定的截止时间或里程碑是什么？
- 有哪些合规或监管约束？
- 预算、基础设施或人力上限是什么？

## 领域与数据

- 哪些实体或类型是核心对象？
- 哪些不变量绝不能被破坏？
- 每类数据由谁负责归属与维护？
- 每个关键字段的唯一事实来源是什么？
- 有哪些保留、审计、删除要求？

## 接口与集成

- 该能力必须与哪些外部系统集成？
- 优先的接口风格是什么（API、事件、内部模块）？
- 需要哪些向后兼容承诺？
- 版本策略是什么？
- 延迟或吞吐的 SLO 要求是什么？

## 可靠性与运维

- 哪些失败模式是不可接受的？
- 需要什么一致性级别？
- 超时、重试、幂等策略是什么？
- 必需的可观测性项有哪些（日志、指标、链路、告警）？
- 哪些条件触发自动或手动回滚？

## 交付与治理

- 哪个团队负责实现？
- 哪个团队负责长期运维？
- 哪些依赖会阻塞交付？
- 发布策略倾向哪种（暗发布、金丝雀、分批）？
- 哪些开放决策需要干系人签字确认？
