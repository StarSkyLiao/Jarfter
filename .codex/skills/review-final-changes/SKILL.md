# 收尾变更审查

## 概述

执行交付前的最终代码审查，聚焦当前本地变更。
仅基于当前本地 Git diff 输出可落地的问题与建议。

## 触发条件与范围

在用户明确表示任务完成、可以收尾时触发。
仅审查当前本地 Git 变更中仍存在的内容（包括 staged 与 unstaged）。
忽略已被用户从本地变更中移除的内容。

## 审查流程

1. 收集当前变更范围。
- 执行 `git status --short`。
- 执行 `git diff --name-only` 与 `git diff --cached --name-only`。
- 结合 `git diff` 与 `git diff --cached` 阅读具体 hunk。

2. 先做潜在 bug 审查。
- 检查正确性风险、回归风险、边界条件遗漏与不安全假设。
- 优先报告会影响运行行为的问题。

3. 条件化执行性能与 GC 审查。
- 仅在明确属于高性能目标的代码路径中执行此项检查。
- 关注热点循环、高频分配、序列化热点与其他潜在 GC 压力来源。

4. 执行 C# 注释规范审查。
- 先读取并遵循：`.codex\skills\csharp-commenting-zh\SKILL.md`。
- 检查 XML 注释完整性、中文注释规范、标点与一致性问题。

5. 执行可读性与可维护性审查。
- 检查命名不清、控制流过于复杂、重复逻辑与难以维护的结构。
- 仅报告对理解成本或维护风险有实质影响的问题。

## 输出规范

先给 findings，并按严重级别排序。
每条问题包含：严重级别、风险说明、文件与行号、修复方向。
若无可操作问题，明确说明“当前本地变更中未发现可操作问题”。
补充静态审查后仍存在的不确定风险或测试缺口。

## 触发示例

- “都完成了，帮我做收尾审查。”
- “准备提交前，按 bug、性能、注释规范、可读性做一次最终 review。”
- “请检查当前本地变更里有没有潜在问题，可以收尾了。”

## 边界

不扩展到与当前目标无关的大规模重构。
不对当前本地 diff 之外的代码提出修改要求，除非存在直接依赖且不改会产生明显风险。
